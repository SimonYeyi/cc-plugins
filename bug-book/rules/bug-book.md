# Bug-Book 规则

## 1. 核心目标
本项目使用 `bug-book` 插件管理 AI 历史错误。**在任何代码修改、功能开发或 Bug 修复任务开始前，必须执行“路径召回”检查**，以防止重复犯错。

## 2. 强制执行流程（静默模式）
当你准备对任何源代码文件（如 `.ts`, `.tsx`, `.js`, `.jsx`, `.py`, `.java`, `.kt`, `.swift`, `.m`, `.dart`, `.cpp`, `.c` 等）进行写入操作时，必须按以下步骤执行：

### 第一步：自动提取路径
从当前上下文（用户指令、报错堆栈、或你刚刚读取的文件）中自动获取目标文件路径。
*   **禁止**询问用户“需要检查哪个文件的路径”。
*   **禁止**在没有明确路径的情况下盲目执行。

### 第二步：调用 Bug-Search 技能召回
利用**第一步提取到的路径**，触发 `bug-search` 技能执行**双重召回**（路径召回 + 影响关系召回）。

### 第三步：基于历史的自我修正
*   **若召回结果为空**：按正常逻辑继续执行任务。
*   **若召回结果不为空**：**不要暂停，也不要询问用户**。将这些历史 Bug 视为对**当前修改逻辑**的“避坑指南”。
    *   *核心原则*：确保你本次对**该文件/该函数**的修改，不会重新引入历史记录中已经修复的错误，也不会触发已知的回归问题。
    *   *强制执行*：**在生成任何代码之前，必须审视召回的 Bug**：
        - **精确匹配**（Bug 涉及的具体函数/代码块正是你要修改的）：**必须**深入分析，问自己：“我现在的修改方案是否会重新触发这个问题？”如果会，立即调整方案。
        - **模糊匹配**（Bug 只是同一个文件，但不同位置/功能）：**快速扫一眼**，确认你的修改与 Bug 描述的功能点无关联即可，无需过度谨慎。
    *   *输出规范*：
        - **一般情况**：无需刻意提及，静默应用教训即可。
        - **高风险情况**（满足任一条件）：**必须**在输出代码前添加警告框，提醒用户注意回归风险：
            - Bug 综合分数 `score >= 30`
            - 存在 `severity >= 7` 的影响关系
        - **利用教训**：如果你的修改方案直接引用了历史 Bug 的解决方案，可以简要说明：“参考 Bug #N 的修复方案...”

## 3. 自动记录新错误（自我进化）
如果在开发过程中你的代码被用户纠正、要求回退，或者你发现了新的逻辑缺陷，**必须立即调用 `bug-record` 技能进行记录**。
*   **原则**：不要等待用户提醒。将“被纠正”视为触发记录的强制信号。
*   **目的**：确保同一个错误不会被犯第二次，实现 AI 的自我进化。

## 4. 为什么这么做？
*   **避免重复劳动**：历史 Bug 往往隐藏在深层逻辑中。
*   **保持代码稳定性**：新功能不应破坏已有的修复成果。
*   **AI 自我进化**：通过不断记录和召回，AI 将越来越熟悉该项目的“坑点”。
